# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The application is violating the strict unidirectional data flow. Processed/calculated output notes (such as octave-shifted Play/Start notes or the snapped Home note) are being pushed into the `activeKeys` Zustand array. This causes the Input Keyboard UI to light up in the Scale Select, Stepper, or Root zones when a user shifts a Play/Start note downward, creating a confusing visual feedback loop.

**Objective:** Audit and refactor `useWebMidi.ts` so that `addActiveKey` and `removeActiveKey` are *strictly and only* called using the raw, physical `note` from the MIDI event. Processed/calculated notes must only be sent to `addOutputKey` and `removeOutputKey`.

## 2. Technical Decisions & Dependencies
* **Unidirectional State Strictness:** The `uiState.activeKeys` array represents physical reality (what the human is pressing). The `uiState.outputActiveKeys` array represents the engine's result. They must never mix.
* **Targeted Excision:** We will remove specific `addActiveKey(targetNote)` and `removeActiveKey(targetNote)` calls in both the Note On and Note Off blocks for the Home Zone and Play/Start Zone in `useWebMidi.ts`.

### Phase 2: Sever Feedback in Note Off Blocks — ✅ Completed
* **Objective:** Remove calculated notes from `removeActiveKey` calls during Note Off events to prevent hung physical keys.
* **Instructions:**

        ```typescript
          freshState.removeOutputKey(targetNote);
          activeNotesRegistry.current.delete(note);
        }
        ```
    4.  Locate the **Play/Start Zone** cleanup block (`note >= 73 && note <= 108`).

        ```typescript
            state.removeOutputKey(targetNote);
          }
          activeNotesRegistry.current.delete(note);
        }
        // Physical key removal remains
        ```
