# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The user has identified a race condition in the `useWebMidi.ts` engine when pressing the A0 (Home) key. Currently, the first press plays the *stale* (previously played) note and hangs, while the second press correctly plays the scale's root note. 

**Root Cause Analysis:** In `useWebMidi.ts`, the Note On handler for A0 (MIDI 21) calls `state.triggerHomeReset()` to mutate the store. Immediately after, it reads `state.activeState.lastPlayedMidi` from the *same* `state` object. Because Zustand's `getState()` returns a frozen snapshot of the state at the time it was called, `state` does not reflect the mutation that just occurred. The engine subsequently maps the stale note into the `activeNotesRegistry` and triggers it. This causes a desync between the actual store state and the active note registry, leading to hung notes during the Note Off cleanup phase.

## 2. Technical Decisions & Dependencies
* **Synchronous State Fetching:** To resolve this without introducing React `useEffect` feedback loops, we must explicitly request a *fresh* state snapshot from Zustand immediately after any synchronous mutation inside the event listener. 
* **Closure Variable Elimination:** To guarantee absolute determinism in the Note Off cleanup, we will stop relying on destructured closure variables like `addActiveKey` and `removeActiveKey` inside the `handleMidiMessage` function. All state access must be routed strictly through `useMidiStore.getState()`.
